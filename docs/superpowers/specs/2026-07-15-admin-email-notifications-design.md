# Email-уведомления администратора о новых заявках — дизайн

## Цель

Уведомлять администратора о каждой успешно сохранённой заявке через короткое email-сообщение. Письмо не должно содержать персональных данных заявителя и должно вести на защищённую страницу этой заявки в админ-панели.

## Область изменений

В первый релиз входят:

- отправка одного письма на адрес администратора, отдельно заданный для каждого окружения;
- ссылка из письма на `/admin/submissions/{id}`;
- отправка только после успешного сохранения заявки в D1;
- конфигурация Cloudflare Email Service для окружений `develop` и `prod`;
- тесты формирования уведомления и сохранения заявки.

В первый релиз не входят повторные попытки доставки, журнал доставок, уведомления заявителю, несколько получателей и изменения интерфейса публичной формы или админ-панели.

## Выбранный подход

Используется нативный Cloudflare Email Service через Worker binding `ADMIN_EMAIL`. Это исключает отдельного почтового провайдера и секретный API-ключ. Binding закрепляет допустимый адрес получателя конфигурацией Cloudflare, поэтому код Worker не сможет отправить уведомление на произвольный адрес.

Очередь Cloudflare и внешние сервисы рассматривались, но не включаются: для одного административного уведомления они не дают преимущества, соразмерного дополнительной конфигурации и сопровождению.

## Поток данных

1. Посетитель отправляет валидную контактную форму.
2. Action `src/routes/+page.server.ts` сохраняет заявку в D1.
3. Репозиторий возвращает ID новой записи.
4. Action формирует абсолютную ссылку из `url.origin` текущего запроса и пути `/admin/submissions/{id}`.
5. Action передаёт отправку email в `platform.context.waitUntil()`.
6. Cloudflare Email Service доставляет письмо на адрес, который задан и разрешён binding-ом `ADMIN_EMAIL` текущего окружения.
7. При сбое отправки Worker логирует ошибку; действие формы всё равно возвращает успех, поскольку заявка уже надёжно сохранена.

## Состав письма

- Отправитель: подтверждённый адрес домена, настроенный в Cloudflare Email Service. Это технический адрес отправителя, а не обязательно существующий почтовый ящик, в котором кто-либо читает папку «Отправленные».
- Получатель: один адрес администратора, заданный только в конфигурации binding-а.
- Тема в `prod`: `Новая заявка #123`, где `123` — ID созданной заявки.
- Тема в `develop`: `TEST: Новая заявка #123`.
- Тело: текст «Поступила новая заявка.» и ссылка «Открыть заявку».

Имя, телефон, email, комментарий, язык и другие персональные данные в письмо не включаются. Страница по ссылке остаётся под существующей защитой Cloudflare Access, поэтому переход из почты не обходит авторизацию.

## Конфигурация окружений

В `wrangler.jsonc` для `develop` и `prod` добавляется binding с одинаковым именем `ADMIN_EMAIL` и параметром `destination_address`. Значения независимы: в `develop` можно указать личный адрес разработчика, в `prod` — адрес клиента. Для изменения получателя достаточно изменить `destination_address` нужного окружения и повторно развернуть Worker; код менять не потребуется.

В `vars` каждого удалённого окружения добавляется несекретное значение `APP_ENV`: `develop` или `prod`. Код выбирает префикс `TEST: ` только при `APP_ENV === 'develop'`; для `prod` префикса нет. Адрес получателя не передаётся в TypeScript и не используется для выбора темы.

Для `local` binding не добавляется: локальная отправка не должна случайно доставлять реальные письма. Код обработки отсутствующего binding-а завершает уведомление без отправки и пишет диагностическую запись в лог.

До развёртывания необходимо в панели Cloudflare Email Service подтвердить домен и адрес отправителя, а также указать адрес получателя для каждого окружения.

## Ошибки и надёжность

- Ошибка D1: заявка и уведомление не создаются; форма возвращает существующую ошибку сервера.
- Отсутствующий binding: заявка сохраняется, Worker фиксирует предупреждение, письмо не отправляется.
- Ошибка Cloudflare Email Service: заявка сохраняется, Worker фиксирует ошибку с ID заявки; пользователь не видит техническую ошибку.
- Повторная отправка не выполняется автоматически. Оператор может открыть заявку в админ-панели; механизм повторов может быть добавлен отдельной задачей через Cloudflare Queue.

## Тестирование и приёмка

- Репозиторий возвращает ID вставленной заявки.
- Логика уведомления формирует тему и ссылку, не включая персональные данные.
- При доступном email-binding отправка передаётся в `waitUntil` только после успешной вставки.
- При отсутствующем binding-е или ошибке отправки action всё равно отвечает успехом после сохранения заявки.
- `npm run check` и `npm test` проходят.
- Ручная проверка в `develop`: новая заявка появляется в админ-панели, администратор получает письмо и ссылка открывает именно эту защищённую заявку.

## Решения, принятые в дизайне

- Вариант: нативный Cloudflare Email Service, без внешнего поставщика.
- Письмо: только факт новой заявки и ссылка.
- Доставка: фоновая, не блокирует успешную отправку формы.
- Получатель: один административный адрес в Cloudflare binding, независимо настраиваемый для `develop` и `prod`.
- Маркировка среды: `TEST: ` в теме только для `develop`.
