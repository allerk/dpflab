# DPFLAB

DPFLAB — двуязычный сайт и приложение для управления контентом сервиса очистки DPF-фильтров в Эстонии. Публичная версия доступна на русском языке по адресу `/` и на эстонском по адресу `/et`; авторизованные сотрудники управляют бизнес-контентом, изображениями и заявками через `/admin`.

Приложение построено на SvelteKit и работает в Cloudflare Workers. Cloudflare D1 хранит редактируемый контент и заявки из формы, R2 — загруженные изображения, а Cloudflare Access защищает административную панель.

## Возможности

- Адаптивный лендинг с метаданными и URL, учитывающими выбранный язык
- Управление FAQ, ценами, контактами, отзывами, сертификатами и примерами «до/после»
- Контактная форма с серверной валидацией и управлением заявками
- Библиотека изображений в R2 и настройка изображений для секций публичной страницы
- Русский и эстонский интерфейсы на базе Paraglide JS
- Аутентификация административной панели `/admin` через Cloudflare Access
- Пятиминутное edge-кеширование публичных GET-ответов; административные маршруты и запросы формы не кешируются
- Раздельные локальное, develop- и production-окружения Cloudflare

## Технологии

- SvelteKit 2 и Svelte 5
- TypeScript и Vite 8
- Tailwind CSS 4 с конфигурацией в CSS
- Paraglide JS 2 для переводов статического интерфейса
- Drizzle ORM и Cloudflare D1
- Cloudflare Workers и R2
- Vitest 4

## Требования

- Node.js 24.15.0 (версия закреплена в `.nvmrc`)
- npm
- Аккаунт Cloudflare и аутентификация Wrangler — только для удалённых операций с базой данных и деплоя

## Локальная разработка

Локальная разработка использует D1 и R2 на базе Miniflare. Локальное окружение не читает и не изменяет ресурсы develop или production.

```bash
nvm use
npm ci
cp .dev.vars.example .dev.vars
```

Укажите email в `.dev.vars`, чтобы настроить локальную учётную запись администратора:

```dotenv
DEV_ADMIN_EMAIL=you@example.com
```

Создайте и заполните локальную базу данных, затем запустите Worker:

```bash
npm run db:migrate:local
npm run db:seed:local
npm run dev
```

Доступные адреса:

- Публичный сайт: `http://localhost:8787`
- Эстонская версия: `http://localhost:8787/et`
- Административная панель: `http://localhost:8787/admin`

Команда `npm run dev` собирает приложение SvelteKit, а затем запускает `wrangler dev --env local`. После изменений, требующих новой сборки, перезапустите команду.

Чтобы сбросить только локальные данные D1 и R2, остановите Worker, удалите `.wrangler/state/v3/d1` и `.wrangler/state/v3/r2`, затем повторно выполните команды миграции и начального заполнения. Операция затрагивает только локальные данные Miniflare.

## Команды

| Команда | Назначение |
| --- | --- |
| `npm run dev` | Собрать и запустить локальный Worker на порту 8787 |
| `npm run build` | Создать production-сборку Cloudflare Worker |
| `npm run preview:develop` | Собрать и запустить локально с привязками develop-окружения |
| `npm run check` | Запустить проверки Svelte и TypeScript |
| `npm test` | Однократно запустить тесты Vitest |
| `npm run test:watch` | Запустить Vitest в режиме наблюдения |
| `npm run db:generate` | Сгенерировать миграцию из `src/lib/db/schema.ts` |
| `npm run db:migrate:local` | Применить миграции к локальной D1 |
| `npm run db:seed:local` | Заменить управляемый seed-данными локальный контент содержимым `seed/seed.sql` |
| `npm run db:migrate:remote` | Применить миграции к production D1 |
| `npm run db:migrate:remote:develop` | Применить миграции к develop D1 |
| `npm run db:seed:remote` | Заполнить production D1 начальными данными |
| `npm run db:seed:remote:develop` | Заполнить develop D1 начальными данными |
| `npm run deploy` | Собрать и развернуть production Worker |
| `npm run deploy:develop` | Собрать и развернуть develop Worker |

Удалённые команды миграции, начального заполнения и деплоя требуют аутентификации Wrangler. Считайте удалённые seed-команды явными операционными действиями: `seed/seed.sql` удаляет и заново создаёт управляемые seed-данными записи FAQ, цен и контактов.

## Архитектура

### Публичное приложение

Paraglide определяет язык из URL. Русский — базовый язык по адресу `/`, эстонский использует префикс `/et`. Файл `src/hooks.ts` сопоставляет локализованные URL с общими маршрутами SvelteKit, а `src/hooks.server.ts` устанавливает `event.locals.locale` и подставляет в документ язык и направление текста.

Публичная страница параллельно загружает контент из D1 и отображает статические секции Svelte вместе с редактируемыми данными. Заявки контактной формы проверяются в `src/routes/+page.server.ts` и сохраняются в `contact_submissions` вместе с языком посетителя и временем создания.

Успешные публичные GET-ответы кешируются в edge-кеше Cloudflare на пять минут с одной минутой `stale-while-revalidate`. Административные URL и запросы, отличные от GET, никогда не кешируются.

### Административное приложение

Административная панель позволяет управлять:

- FAQ и ценами
- отзывами и сертификатами
- контактными данными
- заявками из контактной формы
- элементами галереи «до/после»
- загруженными изображениями R2 и назначением изображений публичным секциям

В развёрнутых окружениях Cloudflare Access пропускает разрешённые учётные записи на edge-уровне. Worker также проверяет JWT из `Cf-Access-Jwt-Assertion`, используя настроенные домен команды и audience приложения. При локальной разработке edge-уровня Access нет, поэтому `DEV_ADMIN_EMAIL` задаёт явную учётную запись разработчика.

### Данные и изображения

Файл `src/lib/db/schema.ts` — единственный источник истины для схемы базы данных. Генерируйте миграции командой `npm run db:generate`, добавляйте полученные файлы из `drizzle/` в Git и применяйте их к каждой целевой базе D1 до деплоя кода, зависящего от новой схемы.

Репозитории получают привязанный к запросу клиент Drizzle, созданный из `event.platform.env.DB`. Worker не применяет миграции во время выполнения.

Загруженные изображения хранятся в R2-бакете соответствующего окружения и доступны по адресу `/images/<filename>`. Поддерживаемые расширения: `.jpg`, `.jpeg`, `.png`, `.webp`, `.gif`, `.svg` и `.avif`; максимальный размер загрузки через административную панель — 10 МБ.

## Интернационализация

В проекте используются два слоя контента:

1. Статический текст интерфейса находится в `messages/ru.json` и `messages/et.json` и компилируется Paraglide в `src/lib/paraglide/` при сборке и разработке. Никогда не редактируйте сгенерированные файлы вручную.
2. Управляемый через административную панель бизнес-контент находится в D1. Переводимые поля FAQ, цен и сертификатов хранятся как JSON-строки с ключами языков; у отзывов язык указан явно.

Чтобы добавить язык:

1. Добавьте его тег BCP 47 в `project.inlang/settings.json`.
2. Создайте `messages/<tag>.json` с полным набором статических сообщений.
3. Добавьте язык в переключатель в `src/lib/components/Header.svelte`.
4. Дополните локализованные seed-данные и контент, управляемый через административную панель.
5. Соберите приложение и проверьте локализованные публичные и административные URL.

## Стилизация

Tailwind CSS настраивается через `src/app.css`; файла `tailwind.config.js` нет. Дизайн-токены объявлены в `@theme` и создают утилиты вроде `bg-accent`, `text-fg-muted` и `rounded-card`.

Пользовательский CSS должен находиться в существующих блоках `@layer base` или `@layer components`, чтобы правила вне слоёв неожиданно не переопределяли утилиты Tailwind. Проект задаёт собственные breakpoint-токены в `src/app.css`; сверяйтесь с ними, а не с настройками Tailwind по умолчанию.

## Тестирование и проверка

Перед созданием pull request запустите стандартный набор проверок:

```bash
npm run check
npm test
npm run build
```

Тесты находятся вне `src/`, в каталоге `tests/`. Тесты репозиториев базы данных используют совместимый с D1 SQLite-клиент в памяти и применяют настоящие файлы миграций.

## Окружения Cloudflare

| Окружение | Worker | База D1 | R2-бакет | Публичный endpoint |
| --- | --- | --- | --- | --- |
| `local` | `dpflab-local` | `dpflab-local` | `dpflab-images-local` | Только локальный Wrangler |
| `develop` | `dpflab-develop` | `dpflab-develop` | `dpflab-images-develop` | `workers.dev` включён |
| `prod` | `dpflab` | `dpflab` | `dpflab-images` | Пользовательский домен; `workers.dev` отключён |

Привязки окружений определены в `wrangler.jsonc`. Файл `.dev.vars` используется локально и игнорируется Git; не добавляйте в репозиторий учётные данные или production-секреты.

При изменении схемы обычный порядок деплоя выглядит так:

```bash
npm run check
npm test
npm run db:migrate:remote:develop
npm run deploy:develop
```

После проверки develop примените production-миграцию до деплоя production:

```bash
npm run db:migrate:remote
npm run deploy
```

Политики Cloudflare Access, пользовательские домены, базы D1 и R2-бакеты должны существовать до того, как соответствующий удалённый деплой станет работоспособным.
